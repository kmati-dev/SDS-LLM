================================================================
เอกสารประกอบการนำเสนอ
Root Cause Analysis — Greedy Speculative Decoding บนภาษาลาว
Corpus: Lao Wikipedia (wikimedia/wikipedia, 20231101.lo)
================================================================

หัวข้อที่วิเคราะห์:
  พฤติกรรมของ greedy speculative decoding (n-gram drafter + greedy
  verifier) เมื่อใช้ Lao Wikipedia เป็น corpus สำหรับการทำนาย —
  drafter เดาถูก/ผิดเมื่อไร, accept ได้กี่ token, speedup เท่าไร,
  และอะไรเป็นตัวขับ/ตัวฉุด

หมายเหตุสำคัญ:
  tokenizer (Qwen 3.5 / Gemma 4) เป็นเพียง "setting ที่ลองสลับ" ตามโจทย์
  ไม่ใช่สิ่งที่นำมาตัดสิน — เรารัน speculative decoding ตัวเดียวกัน
  ภายใต้ tokenizer สองตัว เพื่อดูว่าการเลือก tokenizer ทำให้พฤติกรรม
  spec-decode เปลี่ยนไปอย่างไร

ไฟล์ที่เกี่ยวข้อง:
  กราฟ      : experiments/wiki_lo/artifacts/wiki_lo_part1.png
              experiments/wiki_lo/artifacts/wiki_lo_part2.png
  ข้อมูลดิบ  : experiments/wiki_lo/full_analysis.json
  โค้ด       : analyze_wiki.py
  รันซ้ำ     : /opt/miniconda3/bin/python3.13 analyze_wiki.py --lang lo


================================================================
0. SETUP
================================================================

  Corpus (n-gram DB ของ drafter) : Lao Wikipedia 20231101.lo
                                    4,714 บทความ
  Target (จำลองการ generate)      : 300 บทความ held-out (drafter ไม่เคยเห็น)
  Drafter                         : n-gram + backoff (n -> 1), index แบบ hash
  Verifier                        : greedy เทียบ ground truth
                                    รองรับ depth [1,B] และ width [S,T]
  Metric หลัก                     : speedup (= normal steps / spec steps),
                                    acceptance, step types
  Setting ที่สลับ                  : tokenizer = Qwen/Qwen3.5-4B
                                    หรือ google/gemma-4-31B-it


================================================================
1. ผลรวม SPECULATIVE DECODING  (หัวข้อหลัก)
================================================================

Speedup โดยรวม (n=3, draft budget B=3):  ~1.5–1.6 เท่า
  ภายใต้ Qwen  : 1.51x
  ภายใต้ Gemma : 1.56x
  -> spec-decode บนลาวได้ผลปานกลาง ไม่สูงเท่างาน extractive อย่าง SQuAD
     (1.72x) แต่ดีกว่า baseline ชัดเจน


----------------------------------------------------------------
1.1 Step-type breakdown — drafter พังตรงไหน (n=3, B=3)
----------------------------------------------------------------

  setting        no_draft   full_reject   partial   full_accept
  -----------    --------   -----------   -------   -----------
  (Qwen)            0.4%        71.3%       20.1%       8.6%
  (Gemma)           0.7%        70.7%       17.1%      11.5%

ข้อค้นพบสำคัญที่สุด:
  no_draft ต่ำมาก (< 1%) แต่ full_reject สูง ~71%
  แปลว่า drafter "หา n-gram match เจอเกือบทุกครั้ง แต่เดา token
  ตัวถัดไปผิด"
  => คอขวดคือความแม่นของการทำนาย ไม่ใช่ coverage
     นี่คือ root cause หลักของ spec-decode บนภาษานี้


----------------------------------------------------------------
1.2 Draft budget sweep (n=3, depth) — เดิมพันยาวขึ้นช่วยไหม
----------------------------------------------------------------

  B      speedup(Qwen)   speedup(Gemma)   avg accepted/step
  ---    -------------   --------------   -----------------
  1         1.284x          1.291x             ~0.30
  3         1.509x          1.558x             ~0.55
  6         1.637x          1.712x             ~0.70

  -> budget มากขึ้น speedup เพิ่ม (แบบ diminishing) แต่ full_accept
     ลดลง (จาก ~29% ที่ B=1 -> ~5% ที่ B=6) และ partial เพิ่มขึ้น
     เพราะต้องเดาถูกติดกันหลายตัวยากขึ้น


----------------------------------------------------------------
1.3 N-gram size effect (depth B=4, corpus 1M) — context ยาวขึ้นช่วยมาก
----------------------------------------------------------------

  n      speedup(Qwen)   speedup(Gemma)
  ---    -------------   --------------
  2         1.179x          1.181x
  3         1.568x          1.627x
  4         1.881x          1.853x

  -> n=4 ดีกว่า n=3 ~+20-30% : context ยาวขึ้น = ทำนายแม่นขึ้นชัดเจน
     เป็น lever ที่ได้ผลที่สุด


----------------------------------------------------------------
1.4 Depth vs Width drafting (budget B = S x T เท่ากัน)
----------------------------------------------------------------

  budget          depth (1xB)    best width (S=2)
  ------------    -----------    ----------------
  B=4 (Qwen)        1.573x         1.666x   (+5.9%)
  B=6 (Qwen)        1.637x         1.803x   (+10.1%)
  B=4 (Gemma)       1.633x         1.603x
  B=6 (Gemma)       1.712x         1.729x

  -> width-drafting (กระจายเป็น 2 สาย hedge ที่ branch point) ช่วยได้
     มากในกรณีที่จุดแตกแขนงไม่แน่นอน
     (width-full S x 1 แย่กว่า depth; 2 สายคือจุดดีที่สุด)


----------------------------------------------------------------
1.5 Corpus-size sensitivity — ข้อมูลเยอะขึ้นช่วยไหม
----------------------------------------------------------------

  corpus     speedup(Qwen)   speedup(Gemma)   no_draft
  -------    -------------   --------------   --------
  250k          1.563x          1.611x          ~2%
  1M            1.568x          1.627x          ~1%
  full          1.573x          1.633x          ~0.5%

  -> เกือบแบนราบ : corpus โต 20 เท่า speedup ขยับ ~1% เท่านั้น
     n-gram coverage อิ่มตัวตั้งแต่ 250k token (backoff หาเจอเกือบเสมอ)
     => corpus size ไม่ใช่คันโยก


----------------------------------------------------------------
1.6 ข้อจำกัดของโจทย์ (corpus 1M / 10M / 100M)
----------------------------------------------------------------

  ทั้งภาษาลาวบน Wikipedia มี token รวมเพียง:
    ภายใต้ Qwen  : ~4.9M token
    ภายใต้ Gemma : ~2.9M token
  -> corpus 10M / 100M token สร้างไม่ได้จริงสำหรับลาว (ทำได้แค่ 1M)
     และจาก 1.5 ต่อให้มีก็คงไม่ช่วย


================================================================
2. ผลของการสลับ TOKENIZER  (ส่วน "ลองใช้ tokenizer ต่าง ๆ")
================================================================

รัน spec-decode ตัวเดิมภายใต้ tokenizer 2 ตัว
-> token-based speedup ใกล้กัน (1.51x vs 1.56x)
   ดูเผิน ๆ เหมือนเลือก tokenizer ไหนก็พอกัน — แต่ไม่ใช่
   เพราะ 1 token ของแต่ละตัวแทนข้อความไม่เท่ากัน

ตัวขับเบื้องหลัง:

  รายการ                                    Qwen 3.5    Gemma 4
  ---------------------------------------    --------    -------
  ความยาว token เฉลี่ย (chars/token)            1.06       1.79
  byte-fragment rate (ไบต์ไม่ครบตัวอักษร)        26.4%      0.2%

เพื่อเทียบให้ยุติธรรม จึง normalize speedup เป็น
"อักษรลาวที่ generate ได้ต่อ 1 target-model step":

  (n=3, B=3)                Qwen 3.5    Gemma 4
  -------------------       --------    -------
  token / step (speedup)      1.51x      1.56x
  chars / step                1.51       2.59

  -> พอวัดเป็น "อักษรต่อ step" spec-decode ภายใต้ Gemma เดินหน้าได้
     เร็วกว่า ~1.7 เท่า
     สาเหตุ: Qwen แตกอักษรลาวเป็นไบต์ -> drafter ต้องเดาทีละไบต์
     (ยาก + แต่ละ step ได้ข้อความนิดเดียว); Gemma มี subword ลาวจริง
     -> drafter เดาทีละ "คำ" (แต่ละ accept พ่วงข้อความมากกว่า)


================================================================
3. MISMATCH RCA เชิงคุณภาพ — ทำไม draft ถึงถูก reject (n=3, B=3)
================================================================

ภายใต้ Qwen — reject ระดับ byte / sub-character:
  ctx = '\n\nຄວ'   expected = '�' (byte!)   drafted = 'ເ'
        -> เฉลยเองยังเป็น byte-fragment
  ctx = '\n\n'      expected = 'ຄ'           drafted = 'ร' (อักษรไทย)
        -> สับสนลาว <-> ไทย ผ่าน byte pattern

ภายใต้ Gemma — reject ระดับ "คำ":
  ctx = '\n\n'      expected = 'ຄວາມ' (คำเต็ม)   drafted = 'ราย' (คำไทย)
  ctx = '\n\nຄວາມ'  expected = 'ໝ'              drafted = 'ຫນ'
        -> เสียงเดียวกัน เขียนต่างกัน

  -> ยืนยัน 1.1: drafter หา match เจอแต่เดาผิด และทั้งสอง setting
     มี Lao <-> Thai confusion (สคริปต์ใกล้กัน + คำปนใน corpus)


================================================================
4. สรุป ROOT CAUSE  (เกี่ยวกับ Speculative Decoding)
================================================================

  1. คอขวด = "เดาผิด" ไม่ใช่ "หาไม่เจอ"
     full_reject ~71%, no_draft < 1%
     n-gram backoff หา match ได้เสมอ แต่ continuation มักผิด
     => ขีดจำกัดคือความ predictable ของภาษาลาวภายใต้ tokenization

  2. Corpus size ไม่ใช่คันโยก
     speedup แบนราบ 0.25M–4.9M (coverage อิ่มเร็ว) และลาวมีข้อมูล
     ไม่ถึง 10M token อยู่แล้ว

  3. คันโยกที่ได้ผลจริง:
     (ก) n ใหญ่ขึ้น (n=4 ~ +20-30%)
     (ข) width-drafting (S=2, +6-10% โดยเฉพาะตอน budget สูง)

  4. draft budget ช่วยแบบ diminishing — B สูงได้ speedup เพิ่ม
     แต่ full-accept ยากขึ้น

  5. การเลือก tokenizer เปลี่ยนผลจริงเมื่อวัดให้ถูก —
     token-speedup เท่ากันแต่ chars/step ต่าง ~1.7 เท่า
     (บทเรียนเรื่องการเลือก metric ของ spec-decode ในงานข้ามภาษา/
     ข้าม tokenizer)

  Bottom line:
     บนภาษาลาว n-gram speculative decoding ได้ ~1.5–1.7 เท่า โดยถูก
     จำกัดด้วย "ความแม่นของการเดา" (ไม่ใช่ปริมาณข้อมูล) — ปรับ n และ
     ใช้ width-drafting ช่วยได้ และต้องระวังวัด speedup เป็น "อักษรต่อ
     step" เมื่อเทียบข้าม tokenizer ไม่งั้นจะสรุปผิด
